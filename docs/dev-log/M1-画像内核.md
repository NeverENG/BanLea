# M1 · 画像内核（阶段记录）

> 日期：2026-06-09 ｜ 对应计划 §13 M1
> 状态：**阶段性完成**。types 层已具备 27 维 schema，本次补齐 `core/harness` 的纯逻辑内核，真实 Claude 调用仍需用户 API Key 做 live 验证。

## 本阶段目标（计划 M1）
≥20 维画像 schema；主/子 harness 结构化输出生成；版本化落库。

## 已完成增量一：harness 纯逻辑内核
- 实现 `core/harness` 内核：
  - 主/子画像维度集合选择；
  - 初始画像生成入口；
  - 证据类型到触发维度的确定性推断；
  - 局部重评估 patch schema；
  - patch 合并、画像版本递增、更新时间更新；
  - scope/domain 与跨层维度越界校验；
  - LLM 调用通过 `HarnessModel` 注入，便于无 API Key 的单测。
- 新增 harness 单测：
  - 主/子维度选择；
  - prompt 只暴露当前层允许维度；
  - quiz / 推荐点击等证据触发维度；
  - patch 只覆盖触发维度，其余沿用旧版；
  - 非法跨层维度拒绝；
  - 无证据时不调用模型、不产生新版本；
  - mock LLM 输出合并路径。

## 已完成增量二：画像与证据仓储层
- 实现 `db` 仓储层：
  - `SqlExecutor` 抽象，隔离 Tauri SQL 运行时，便于 Node/Vitest 单测；
  - `portraitRepo`：保存画像版本、读取最新版本、按版本读取、按 domain 列表读取、计算下一版本号；
  - `evidenceRepo`：写入证据、读取未消费证据、按 domain 读取证据、批量标记 `consumed_in_version`；
  - `db/index.ts` 统一导出 repo 与真实 Tauri DB 单例。
- 新增仓储单测：
  - 验证 `portrait_versions` 写入字段、JSON 解析和 snake_case 映射；
  - 验证 `nextVersion` 对已有/新 domain 的版本递增；
  - 验证 `evidence` 写入、payload 序列化/解析、未消费查询；
  - 验证证据空 id 不写库、非空 id 批量标记消费。

## 已完成增量三：harness 编排服务
- 实现 `runHarnessUpdate`：
  - 读取指定 domain 的最新画像版本；
  - 拉取未消费证据；
  - 无证据时跳过，避免无意义 LLM 调用和空版本；
  - 无历史画像时走初始画像生成；
  - 有历史画像时走局部重评估；
  - 保存新画像版本；
  - 将参与本次画像更新的证据标记为 `consumed_in_version`。
- 新增编排单测：
  - 无未消费证据时不调用模型、不写库；
  - 首次建档生成画像并消费证据；
  - 已有画像时局部重评估、递增版本并消费证据。

## 已完成增量四：画像重评估触发器
- 实现触发策略配置：
  - `minEvidenceCount`：累计未消费证据达到阈值再重评估；
  - `strongFeedbackDwellSeconds`：推荐点击长停留视为强反馈；
  - `lowQuizScore`：测验低分视为卡点/误区需要及时更新。
- 实现 `shouldTriggerHarnessUpdate`：
  - 无证据跳过；
  - 无历史画像且有证据时触发首次建档；
  - 证据数达到阈值时触发；
  - 推荐点击长停留或跳过时触发；
  - 测验低分时触发；
  - 证据不足且无强信号时继续积累。
- 新增触发器单测，覆盖上述全部分支。
